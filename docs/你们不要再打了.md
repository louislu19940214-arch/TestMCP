## 开场白：同一个界面，三种“真相”

你见过的剧本大概是这样的：

- 美术同学：我这套 UI 已经很完美了，PS 里就是这样。
- 前端同学：我这边也很完美了，游戏里就是这样。
- 观众（PM/策划/老板/玩家）：你们两个能不能先别吵，为什么按钮在飘？

于是大家开始对线：  
“你的切图不对。”  
“你的锚点不对。”  
“你的字体不对。”  
“你的间距不对。”  
“你的眼睛不对。”  

然后就没有然后了——因为每次都靠手搓复原，误差是必然的，吵架是高频的，返工是稳定的。

这份文档就是一个小小的尝试：**让 PSD 到 Prefab 这段路尽量可复现、可对齐、可追责（并尽量少靠吼）**。

---

## 这到底想解决什么问题

我们关心的不是“你觉得像不像”，而是：

- **误差从哪里来**：坐标基准不统一、图层层级不清晰、资源名对不上、规则靠人记。
- **怎么让误差变少**：把“人脑猜测”换成“机器执行”，把“口头约定”换成“落地规则”。
- **怎么让大家不再打**：让每一次产出都有可追溯的中间产物（JSON/规则），能复跑、能比对、能迭代。

一句话概括：

> **用一个中间格式（UI Spec JSON）把 PSD 的结构和坐标固定下来，再通过 Cocos MCP 把它变成 Prefab。**

---

## 我们的“停战协议”（项目的基本思路）

### 1) 美术同学只要做到一件事：命名可对齐

只要满足一个朴素到过分的条件，后面事情会顺很多：

- **像素层/背景图层名 = 导出图片文件名（不含扩展名）**

比如资源是 `common_btn_red.png`，图层名就叫 `common_btn_red`。  
（不要发明“红色按钮最终版_v12_真的不改了”，求你。）

为什么？因为当前流程会用这个名字去找贴图，再通过 `.meta` 拿到 **SpriteFrame 的 uuid**，从而避免“紫块”与手工拖拽绑定。

### 2) 前端同学也只要做到一件事：别手搓，跑流水线

日常工作最推荐的姿势不是“复原 UI”，而是：

- **把 PSD 解析成 `assets/ui-spec/*.json`**
- （可选）写 `*.rules.json` 让组件自动上身（Button、Sliced、Label……）
- **通过 MCP 把 JSON 生成 Prefab**

这样每次变更可复跑、可对比、可复用，不需要“凭经验摆像素”。

---

## 三段式流程（从 PSD 到 Prefab）

这一段是“硬菜”，但我们保证它尽量像剧透一样短。

### A. PSD → UI Spec（JSON / 可选 MD）

在 `tools/` 下执行脚本，输出一个中间产物：

- `assets/ui-spec/你的界面.json`
- （可选）`assets/ui-spec/你的界面.md`

这个 JSON 的意义是：**把 PSD 的层级、尺寸、位置、文本等信息固定为可读、可版本化的描述**。

### B.（可选）加规则：让“经验”变成“配置”

同目录同名放一个：

- `assets/ui-spec/你的界面.rules.json`

它的意义是：**按命名/类型把组件自动补齐**（例如图层名包含 `btn` 就挂 `cc.Button`）。

小提醒：JS 正则不支持 `(?i)`，要用 `flags: "i"`（这条在旧日战场上牺牲过很多人）。

### C. UI Spec → Prefab（MCP 出场）

打开 Cocos 工程，启动 `cocos-mcp-server`，然后让脚本通过 MCP 调用编辑器能力生成 Prefab。

它的意义是：**让 Prefab 的创建不再依赖人肉点选**，并且把“怎么生成”的责任链收敛到：JSON + 规则 + 资源目录。

---

## 为什么它能“少一点误差”

不是因为我们突然变强了，而是因为我们把误差来源从“人”换成了“规则”：

- **统一坐标基准**：解析阶段固定 `--origin`（团队内统一 `center` 或 `topleft`）。
- **统一资源绑定方式**：通过 `.meta` 找 SpriteFrame uuid，减少“拖错/漏绑/临时替换”的漂移。
- **统一组件推断**：把“看起来像按钮所以我加了 Button”变成规则命中。
- **统一可追溯产物**：JSON/规则在版本库里，能复盘“为什么变成这样”。

我们不承诺 100% 魔法复刻（复杂文本效果、渐变描边等仍可能需要降级策略），但它能显著减少“每次都从零复原”的不可控成本。

---

## 交付给团队的“最小承诺”

如果你只是来体验一下，不想背一身工程学债务，这个项目想做到的最低门槛是：

- **美术**：按命名规范交 PSD + 贴图（导入到工程，`.meta` 存在）。
- **前端**：能跑 `tools/` 的脚本，Cocos 里能启动 MCP。
- **产物**：你能得到一个可用 Prefab，并且下一次改 PSD 你能复跑得到“同样逻辑”的 Prefab。

---

## 这只是“抛砖引玉”：欢迎继续开发可能性

当前是一套可跑通的 MVP。接下来如果你愿意一起把这出偶像剧拍成系列，可以考虑这些方向：

- **更强的规则系统**：九宫格、自动 Layout、富文本、交互组件一键上身。
- **更稳定的对齐策略**：字体替换策略、分辨率适配（多套设计稿）、像素对齐与安全区。
- **更完整的产物链路**：JSON diff、自动截图对比、CI 里跑校验（“不通过就不准合并”那种）。
- **从“切图依赖”走向“资源映射”**：图层到资源的映射表、缺图提示、自动导出（可选）。

如果你看完觉得“这玩意儿有点意思”，那就太好了：  
**欢迎你来补一块砖**——让大家以后少打一场架，多合一次 prefab。
